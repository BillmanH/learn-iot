# SASL/SAS Authentication vs System Assigned Managed Identity — Review

## The Question

The original assumption was that the AIO-to-Fabric RTI dataflow would use **System Assigned Managed Identity (SAMI)** for authentication, leveraging the Arc + Workload Identity Federation (WILF) stack that's already in place. The "sales pitch" is:

> Because of Arc and WILF, you can use Entra ID to secure the dataflow connection — no certificates, no shared secrets, one unified identity system.

Instead, the repo currently uses **SASL/Plain with SAS connection strings stored in Key Vault** for the Fabric RTI Kafka endpoint. This document captures what we know, what the gap is, and potential paths forward.

## The Gap: Why SASL Instead of Managed Identity?

### What the architecture supports (in theory)

The full WILF chain is already set up:

1. **OIDC Issuer** enabled on Arc cluster (`oidcIssuerProfile.enabled = true`)
2. **Workload Identity** enabled (`securityProfile.workloadIdentity.enabled = true`)
3. **AIO instance has a System Assigned MI** with `principalId`
4. **Arc cluster has a System Assigned MI** with identity
5. **Event Hubs Data Sender/Receiver** roles are granted to both identities at the RG scope (`grant_entra_id_roles.ps1`)

This chain works for **Azure Event Hubs** — confirmed. AIO can authenticate to an Event Hub endpoint using `SystemAssignedManagedIdentity` with no secrets.

### Where it breaks: Fabric RTI Event Stream Custom Endpoint

The Fabric RTI custom endpoint exposes a **Kafka-protocol** ingestion surface. The issue is:

1. **Fabric Event Stream custom endpoints currently only offer SAS key authentication in the Kafka protocol UI**. When you create a custom endpoint in Fabric and choose Kafka protocol, the only authentication option shown is "Shared access key (SAS)". There is no option for "Managed Identity" or "Microsoft Entra ID" in the Fabric Event Stream custom endpoint configuration.

2. **The Kafka bootstrap server for Fabric** (`*.servicebus.windows.net:9093`) sits behind Fabric's Event Stream service, NOT a regular Azure Event Hub. Even though the wire protocol is Kafka and the domain looks like Service Bus / Event Hubs, the authorization plane is Fabric-specific. Fabric controls the access policies, and the SAS key is generated by Fabric (not by Azure Event Hubs ARM).

3. **RBAC roles like `Azure Event Hubs Data Sender` don't apply to Fabric-managed Event Streams**. Those roles work on Event Hub namespaces that YOU own in your subscription. The Fabric Event Stream's underlying Event Hub namespace is Fabric-managed (internal to Fabric's infrastructure) — your subscription can't assign RBAC roles to it.

### The root cause

**Fabric Real-Time Intelligence Event Stream custom endpoints do not support Microsoft Entra ID / Managed Identity authentication for external clients as of early 2026.** The only authentication method available for Kafka-protocol custom endpoints is SAS key-based (connection strings).

This is a Fabric-side limitation, not an AIO limitation. AIO's dataflow engine fully supports `SystemAssignedManagedIdentity` for Kafka endpoints (it works with standard Azure Event Hubs). But because Fabric doesn't expose its Kafka endpoint with Entra-based auth, we can't use it.

## Impact

### What the SAS approach requires (extra operational burden)

1. **Key Vault secret management** — connection string must be stored in Key Vault and synced to the cluster via CSI driver
2. **Secret rotation** — SAS keys have expiry; they must be manually rotated in both Fabric and Key Vault
3. **SecretProviderClass (`aio-akv-sp`)** — additional K8s CRD that must exist and be configured correctly
4. **Secret sync managed identity** — a User Assigned MI (`<cluster>-secretsync-mi`) must exist with `Key Vault Secrets User` role
5. **OIDC + WILF must still be enabled** — ironically, you need the full WILF stack working even for secret sync to function
6. **Error-prone secret reference format** — the `aio-akv-sp/<secret-name>` format is unintuitive and a frequent source of the "secret management is not configured" error (see `docs/VERIFY_SECRET_MANAGEMENT.md`)

### What managed identity would eliminate

- Items 1-4 above (Key Vault, secret rotation, SecretProviderClass, secretsync MI)
- The entire `setup-SAS-for-RTI.sh` script
- The Key Vault-related sections of `Deploy-FabricEndpoint.ps1`
- Most of the troubleshooting in `VERIFY_SECRET_MANAGEMENT.md`
- The need for `fabric-connection-string` in Key Vault

## Is There a Bug Filed?

**No bug has been filed in this repo or tracked externally for this issue** (as far as I can find in the codebase). The README_ADVANCED.md comment `# Not SystemAssignedManagedIdentity for Fabric` acknowledges the limitation but doesn't reference a tracking item.

### Recommended actions

1. **File a Fabric UserVoice / feedback item** requesting Entra ID / Managed Identity support for Event Stream custom endpoints (Kafka protocol). This is the primary blocker.

2. **File/check for an AIO documentation issue** — the AIO docs should clearly state that `SystemAssignedManagedIdentity` works for Event Hubs but NOT for Fabric RTI custom endpoints, and explain why.

3. **Check Fabric roadmap** — Fabric has been rapidly evolving. Managed Identity support for Event Stream custom endpoints may already be on the roadmap or in preview.

## Potential Workarounds / Alternatives

### Option A: Use Azure Event Hubs as an intermediary (Managed Identity works)

```
AIO MQTT → Dataflow → Azure Event Hub (SAMI auth) → Fabric Event Stream (Event Hub source)
```

- AIO authenticates to a standard Azure Event Hub namespace using `SystemAssignedManagedIdentity` — this works today
- Fabric Event Stream adds the Event Hub as a source (built-in integration, no SAS needed on the Fabric side)
- **Pro**: Full managed identity chain, no secrets, no Key Vault for this flow
- **Con**: Extra hop (Event Hub in between), additional Azure resource cost, slight added latency

### Option B: Use Fabric Event Stream's Event Hub-compatible source directly

If the Fabric Event Stream can consume from an Azure Event Hub (which it can), you could:

1. Create an Azure Event Hub namespace in your subscription
2. AIO dataflow sends to Event Hub using SAMI (confirmed working)
3. Fabric Event Stream subscribes to that Event Hub
4. Eliminates SAS entirely from the pipeline

### Option C: Wait for Fabric to support Entra ID on custom endpoints

- Monitor Fabric release notes
- When available, switch `fabric-endpoint.yaml` from `Sasl` to `SystemAssignedManagedIdentity`
- Remove Key Vault secret, secret sync, and related infrastructure

### Option D: Use Fabric OneLake / Direct Lake instead of Event Stream

- AIO dataflow writes to Azure Storage (SAMI works) or ADX (SAMI works)
- Fabric uses Direct Lake / OneLake shortcuts to read the data
- Eliminates the Kafka endpoint entirely
- **Con**: Not real-time streaming; batch or near-real-time only

---

## Recommendation

**Option B (Event Hub intermediary)** is the cleanest near-term path:

- Eliminates all SAS/Key Vault plumbing for the RTI dataflow
- Uses the proven `SystemAssignedManagedIdentity` auth that already works
- Fabric has first-class Event Hub source support
- The `grant_entra_id_roles.ps1` script already assigns `Event Hubs Data Sender/Receiver` roles
- Minimal code changes: swap `endpointType: Kafka` with Fabric's bootstrap → `endpointType: Kafka` with your own Event Hub

However, this adds an Azure resource (Event Hub namespace) to the architecture and a small amount of additional cost.

**Long-term**: Monitor Fabric for Entra ID support on custom endpoints and switch to direct SAMI auth when available.

## Discussion Points

1. **Is Event Hub intermediary acceptable?** Adds cost (~$11/month for Basic tier) but eliminates all secret management complexity.
2. **Should we implement Option B now?** The WILF/SAMI infrastructure is already deployed and tested. We could have an Event Hub-based flow working quickly.
3. **Should we file a Fabric feature request?** If Entra ID support for custom endpoints is not already tracked.
4. **Dual-path?** Keep the SAS path documented as a fallback while implementing the Event Hub path as the "recommended" approach.
