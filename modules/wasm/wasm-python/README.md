# wasm-python - Factory OEE Enrichment (Python)

Python WASM module that enriches factory telemetry messages with OEE metrics.
This is the Python counterpart to `modules/wasm/wasm-rust`.

Both modules produce identical output under the `_aio` key, but the Python version
adds `"processor": "wasm-py"` so you can tell them apart in downstream analytics.

---

## What this module does

Reads any `factory/#` MQTT message from the edgemqttsim, computes OEE scores,
and adds an `_aio` object to every message:

```json
{
  "machine_id": "cnc-001",
  "status": "running",
  "cycle_time": 115,
  "_aio": {
    "oee": 0.9217,
    "availability": 1.0,
    "quality": 1.0,
    "performance": 0.9217,
    "alert_level": "ok",
    "processor": "wasm-py",
    "sdk_version": "componentize-py-0.14"
  }
}
```

## ACR Task (recommended - no local toolchain needed)

```powershell
cd modules/wasm/wasm-python
az acr run --registry testaioacr --file acr-task.yaml .
```

This runs three steps in Azure:
1. `build-wasm`  - compiles oee_enrich.py using the official Python WASM builder; output at `bin/x86_64/release/oee_enrich.wasm`
2. `push-wasm`   - pushes factory-transform-wasm-py:latest to ACR
3. `push-graph`  - pushes factory-transform-graph-py:latest to ACR

## Local build (if you want to build on your machine)

Requirements:
- Python 3.8+
- componentize-py 0.14
- WIT schema files from the explore-iot-operations repo

```bash
pip install "componentize-py==0.14"

# Get the WIT schema files (all must be present together)
git clone https://github.com/Azure-Samples/explore-iot-operations.git
cp -r explore-iot-operations/samples/wasm-python/schema ./schema

# Generate Python bindings (creates map_impl/ directory)
componentize-py -d ./schema -w map-impl bindings ./

# Compile to WASM
componentize-py -d ./schema -w map-impl componentize oee_enrich -o oee_enrich.wasm

# Verify
wasm-tools component wit oee_enrich.wasm   # should show map interface export
```

Push manually after building locally:
```bash
oras push testaioacr.azurecr.io/factory-transform-wasm-py:latest \
  --artifact-type application/vnd.module.wasm.content.layer.v1+wasm \
  --disable-path-validation \
  bin/x86_64/release/oee_enrich.wasm:application/wasm

oras push testaioacr.azurecr.io/factory-transform-graph-py:latest \
  --config /dev/null:application/vnd.microsoft.aio.graph.v1+yaml \
  graph.yaml:application/yaml \
  --disable-path-validation
```

## Deploy to the cluster

```bash
kubectl apply -f dataflow-graph.yaml

# Check it loaded
kubectl get dataflowgraph -n azure-iot-operations
# factory-oee-graph-py should appear

# Watch for errors
kubectl describe dataflowgraph factory-oee-graph-py -n azure-iot-operations
```

It also appears in the AIO portal under Dataflows -> Graphs as
"Factory OEE Transform (Python)".

## Test end-to-end

Subscribe to the transformed output topic (from inside the cluster):
```bash
mosquitto_sub -h aio-broker -t "factory/transformed-py" -v
```

Publish a test message (simulating edgemqttsim):
```bash
mosquitto_pub -h localhost -t "factory/cnc" \
  -m '{"machine_id":"cnc-001","status":"running","cycle_time":130,"quality":"good"}'
```

Expected output includes `_aio.processor = "wasm-py"`.

## Key differences from the Rust version

| Aspect         | Rust (modules/wasm/wasm-rust)      | Python (this module)                         |
|----------------|------------------------------------|----------------------------------------------|
| Build tool     | cargo + wasm_graph_sdk             | componentize-py                              |
| ACR build step | rust:1.85 container                | python-wasm-builder container                |
| Output file    | target/wasm32-wasip2/release/*.wasm| bin/x86_64/release/oee_enrich.wasm           |
| Module size    | ~2 MB                              | ~5 MB                                        |
| ACR artifact   | factory-transform-wasm:latest      | factory-transform-wasm-py:latest             |
| Graph artifact | factory-transform-graph:latest     | factory-transform-graph-py:latest            |
| K8s resource   | factory-oee-graph                  | factory-oee-graph-py                         |
| Output topic   | factory/transformed                | factory/transformed-py                       |
| Processor tag  | (no tag)                           | _aio.processor = "wasm-py"                   |

The `oras push` artifact types, `dataflow-graph.yaml` structure, and AIO portal
integration are identical between the two versions.

## Common build errors

| Error                                    | Cause                    | Fix                                                    |
|------------------------------------------|--------------------------|--------------------------------------------------------|
| componentize-py can't find WIT files     | Missing schema directory | Use -d flag with path to directory containing all .wit |
| componentize-py version mismatch         | Stale generated bindings | Delete map_impl/ and regenerate with same version      |
| wasm-tools component check fails         | Wrong target             | Must use wasm32-wasip2, not wasm32-wasi                |
| DataflowGraph stuck in Pending           | Artifacts not in ACR     | Run acr-task.yaml first, check both tags exist         |
| Module not visible in portal             | Missing --artifact-type  | ORAS push requires the exact --artifact-type value     |
